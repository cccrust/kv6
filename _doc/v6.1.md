# kv6 v6.1 規劃文件

## 版本資訊

- **版本**: 0.1.0
- **前置版本**: kv6 v0.1.0 (fork from kv5)
- **目標**: 引入 LSM-Tree 持久化儲存

## 概述

將目前的記憶體儲存（DashMap + JSON 快照）替換為 LSM-Tree 架構，提供高效寫入與持久化能力。

## 架構目標

```
Write Path:
  PUT -> MemTable (SkipList) -> Flush -> SSTable (L0) -> Compaction -> SSTable (L1+)

Read Path:
  MemTable -> Bloom Filter -> SSTable (L0) -> SSTable (L1) -> ...
```

## 實作重點

### 1. MemTable (SkipList)

- 使用跳表實現記憶體中的有序儲存
- 支援並發讀寫
- 到達容量閾值時觸發 flush

### 2. SSTable (磁碟檔案)

- 自訂二進位格式
- 包含: data block, index block, bloom filter
- Key 排序儲存於磁碟

### 3. Bloom Filter

- 每個 SSTable 附加 Bloom Filter
- 快速判斷 Key 是否存在於檔案
- 減少不必要的磁碟 I/O

### 4. Compaction (背景壓縮)

- 非同步執行，不阻塞前台讀寫
- L0 -> L1 層級合併
- 合併時清除過期資料

### 5. 相容性

- 保持 Redis 相容 API
- 平滑遷移：先 hybrid 模式，再逐步移轉
- 保留 JSON 匯入匯出工具

## 目錄結構

```
src/
├── lsm/
│   ├── mod.rs      # LSM-Tree 入口
│   ├── memtable.rs # SkipList 實現
│   ├── sstable.rs  # SSTable 格式
│   ├── builder.rs  # SSTable 建構器
│   ├── reader.rs   # SSTable 讀取
│   └── compaction.rs # 壓縮邏輯
└── store/
    ├── store.rs    # 現有 Store
    └── lsm_store.rs # 新 LSM Store (或整合)
```

## 實作順序

1. **Phase 1**: SkipList MemTable 基礎實作
2. **Phase 2**: SSTable 寫入與讀取
3. **Phase 3**: Bloom Filter 整合
4. **Phase 4**: Compaction 背景任務
5. **Phase 5**: 與現有 Store 整合

## 測試策略

- 單元測試：SkipList, SSTable, Bloom Filter
- 整合測試：Write -> Flush -> Read 流程
- 效能測試：寫入吞吐量、查詢延遲
- 相容性測試：現有 Redis 相容命令
